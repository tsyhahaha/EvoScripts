### 1. 系统定位与核心价值

本工具是一个基于“测试驱动开发 (TDD) + 数据驱动迭代”的专用代码智能体调度系统。其核心目的是将开发者从繁琐的长尾数据清洗逻辑中解放出来。特别是在准备自然语言处理 (NLP) 训练集或解析大规模 JSONL 结构化数据时，利用底层 Code Agent 配合严格的统计学评估机制，自动生成、验证并优化高精确率的 Python 清洗脚本。

### 2. 系统架构模块定义

系统在 MVP 阶段分为三个核心且解耦的模块：

| 模块名称 | 核心职责 | 技术选型建议 |
| :--- | :--- | :--- |
| **主控引擎 (Orchestrator)** | 负责解析需求、管理模板上下文、计算抽样数量、调度底层的执行环境与评判逻辑。 | Python (主控脚手架脚本) |
| **底层执行器 (Code Agent)** | 负责接收主控引擎的 Prompt 和模板，在受控沙盒内编写 `cleaner_v1.py`、执行代码并修复基础的语法报错。 | Claude Code CLI 或 OpenDevin |
| **评判引擎 (LLM Judge)** | 依据对齐后的 Rubric，对清洗输出的样本进行盲评，产出包含 Good/Bad 判定及理由的审查报告。 | GPT-4o 或 Gemini 1.5 Pro API |

---

### 3. 核心工作流 (Core Workflows)

MVP 阶段包含两个串行运行的生命周期阶段。

#### 阶段一：品味对齐 (Taste Alignment)
此阶段的目标是解决“缺乏 Ground Truth”的问题，将用户的自然语言需求转化为机器可严格执行的评分规则。

1. **接收输入**：载入业务数据源样本、自然语言清洗需求、以及用于规范 API 请求或并发写法的参考代码模板。
2. **微量抽样**：主控引擎从原始数据中随机抽取极少量样本（例如 3-5 条）。
3. **规则起草**：评判引擎根据自然语言需求，生成一份结构化的 Judge Rubric（评分标准），并以此对抽样数据进行初步“盲评”。
4. **人工确认 (HITL)**：将 Rubric 和初步盲评结果输出至终端，要求用户进行复核。
5. **规则锁定**：用户可通过自然语言微调 Rubric（如补充特定边界条件）。一旦确认，该 Rubric 被冻结，作为后续自动循环的唯一真理。

#### 阶段二：自动化进化循环 (Automated Evolution Loop)
此阶段的目标是驱动 Code Agent 不断修改清洗规则，直至被过滤出的目标数据准确率满足要求。

1. **起草脚本**：主控引擎将“需求 + 锁定后的 Rubric + 参考模板”打包发给 Code Agent，要求生成初版脚本。
2. **小批量抽样试跑**：每次从原始数据中随机抽取 30 条样本，Code Agent 将脚本应用于这些样本进行过滤，将其分为“目标数据”和“非目标数据”两类。（注：若遇语法错误，Code Agent 自行基于 Error Traceback 修复，不计入进化大循环）。
3. **自动化打分**：评判引擎依据冻结的 Rubric，重点对被标记为“目标数据”的样本进行审查（检验 False Positive/误收），同时也应当适量抽样或全量审查被标记为“非目标数据”的样本（检验 False Negative/误杀）。
4. **反思与进化**：
    * 若“目标数据”里混杂了非目标数据，或“非目标数据”中遗漏了真实目标数据，主控引擎将这些 Bad Cases 及其出错原因打包，发回给 Code Agent 引导其优化清洗规则（或基于错误反思完善 Rubric）。返回步骤 2。
    * 触发人工干预周期（如 `every_n_loops=2`），可在终端挂起并询问用户是否继续。
5. **循环终止**：若在这 30 条原样本的过滤与打分结果中，不仅“目标数据”的准确率 (Precision) 达标（如 90%），并且“非目标数据”的误杀率控制在极低水平（提升 Recall），循环安全终止并输出最终脚本。

---

### 4. 固定抽样与退出条件

为了兼顾执行效率与校验准确性，系统采用小批量固定抽样策略：
每次对原始数据随机抽取 30 条进行试跑。评判引擎关注“目标数据”以监控 Precision，并适度抽样“非目标数据”以监控 Recall。
**退出条件**：由 LLM Judge 盲评验证后，这批被划分为“目标数据”的样本其真正例（True Positive）比例需达到或超过 90%，同时误杀的样本数在可容忍范围内。


---

### 5. 异常处理与系统边界

* **沙盒隔离限制**：MVP 阶段不对提取的代码模板进行永久化（不写入外部 SDK），仅作为单次会话的上下文供 Code Agent 组装使用。
* **空集/低召回预警**：如果 Code Agent 生成的规则过于严格，导致 30 条样本过滤后没有任何或极少“目标数据”（输出为 0 或极低占比），主控引擎应立刻阻断，并将“召回率过低/全部被误杀”作为异常 Feedback 打回给 Code Agent 要求其放宽规则。
